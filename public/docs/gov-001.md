## 1. Purpose and Scope

This framework establishes the regulatory requirements for standardizing multi-site neural data sharing in epilepsy research. "Neural data" covers all modalities used in epilepsy investigation including structural MRI, functional MRI, MR angiography, perfusion (ASL), CT, DWI, field maps, scalp EEG, and intracranial EEG. The framework is the authoritative source from which all SOPs, tools, and onboarding processes are derived. No operational deliverable shall be finalized without traceability to specific requirements in this document.

**Scope:** All multi-site neural data shared in epilepsy research across institutions, covering 11 scan types (T1w, T2w, FLAIR, MR angiography, CT, DWI, ASL, fMRI, field maps, scalp EEG, iEEG) organized into 8 BIDS modality folders across three clinical sessions (pre-implant, post-implant, post-surgery). The framework defines how data must be structured, validated, and de-identified before sharing through cloud and on-premise standardized data infrastructure toward building a learning health system infrastructure. The choice of specific infrastructure (Pennsieve, institutional cloud, on-premise archive, etc.) is left to each site. The framework is grounded in four regulatory pillars: FAIR Data Principles, ALCOA+ Data Integrity, HIPAA/PHI Protections, and the NIH Data Management and Sharing Policy (2023), supplemented by Quality Management System (QMS) practices.

**Validation approach:** The framework will first be validated internally at Penn, then externally with one partner site before broader adoption.

## 2. Regulatory Requirements

### 2.1 FAIR Data Principles

The FAIR principles (Wilkinson et al., 2016) are operationalized as follows:

| Principle | Requirement | Implementation |
|---|---|---|
| Findable | Persistent IDs; rich standardized metadata | BIDS folder naming; data infrastructure metadata; REDCap subject registry |
| Accessible | Standardized retrieval with authentication | Platform-level access controls; authenticated retrieval |
| Interoperable | Formal vocabularies; cross-dataset compatibility | BIDS v1.8+; NIfTI/EDF/BrainVision formats; JSON sidecars |
| Reusable | Provenance; community standards for reuse | Acquisition params in JSON; participants.tsv; CHANGES file |

### 2.2 ALCOA+ Data Integrity

All data must meet ALCOA+ standards. Key requirements and verification methods:

| Principle | Requirement | Verification |
|---|---|---|
| Attributable | Each upload linked to identified contributor; REDCap records who/when | Sharing platform upload logs; REDCap audit trail |
| Legible | Standardized formats (NIfTI, EDF, JSON); BIDS naming; no ambiguous abbreviations | BIDS validator; manual review |
| Contemporaneous | Acquisition timestamps preserved in JSON sidecars | JSON sidecar audit |
| Original | Source DICOMs retained; BIDS files are verified copies with documented conversion | dcm2niix logs; checksums |
| Accurate | BIDS validation passes; metadata matches acquisition parameters | BIDS validator report |
| Complete | All sessions have required modalities; all JSON fields populated; participants.tsv complete | Completeness checklist |
| Consistent | Identical naming conventions and metadata definitions across all sites | Cross-site audit |
| Enduring | Long-term storage on the site's chosen data infrastructure; backup procedures; retention per NIH requirements | Storage verification |
| Available | 24/7 platform access; API retrieval; offline backup at the originating site | Access verification |

### 2.3 HIPAA / PHI Protection

All 18 HIPAA identifiers must be removed before data leaves the originating institution. Neuroimaging-specific PHI risks and required actions:

| Data Element | PHI Risk | Required Action | Verification |
|---|---|---|---|
| DICOM headers | Patient name, DOB, MRN | Strip all identifiers during dcm2niix conversion | Post-conversion header audit |
| File names | May contain patient names/MRNs | Replace with BIDS `sub-<ID>` (no PHI linkage) | Naming convention review |
| Facial features | 3D reconstruction from T1w/T2w | Apply defacing (pydeface/mri_deface) | Visual inspection |
| Dates | Exact dates may identify individuals | Shift or remove; retain relative timing only | JSON sidecar date audit |
| EEG/iEEG headers | Patient name in recording ID | Replace with coded subject ID | Header field review |
| REDCap data | Demographics, clinical history | Age ranges only; coded IDs; no direct identifiers | Data dictionary review |

**Subject ID key management:** The key linking coded IDs to real patient identifiers is stored only at the originating institution in a secure, access-controlled system. It is never uploaded to any data infrastructure or shared externally. Access is limited to the site PI and designated data manager.

### 2.4 NIH Data Management and Sharing Policy (2023)

This framework ensures compliance with the NIH DMSP requirements: data types/formats specified (Section 3), metadata standards identified (Section 4), preservation on the site's chosen long-term data infrastructure, sharing timeline aligned with publication/grant end, access controls via platform permissions and data use agreements, and oversight through roles (Section 5) and compliance monitoring (Section 7).

### 2.5 Quality Management System Practices

- **Version Control:** Major.minor format. Major for structural changes; minor for edits.
- **Document Numbering:** Unique IDs (GOV-001, SOP-BIDS-001, SOP-PENNSIEVE-001, SOP-REDCAP-001).
- **Change Tracking:** All changes logged in VERSION_LOG.md with date, author, and summary.
- **Review/Approval:** Governance framework changes require advisor review. SOP changes require governance lead review.
- **Training Records:** Site personnel must complete onboarding (ONBOARD-001) before sharing data through the initiative.

## 3. Data Standards by Modality

All standards derive from BIDS v1.8+ and are supplemented by FAIR, ALCOA+, and HIPAA requirements. Each modality requires a BIDS validator pass, complete JSON sidecars, and de-identification before upload.

| Modality | Session(s) | Format | Naming Example | Key Metadata (JSON) | De-ID Notes |
|---|---|---|---|---|---|
| MRI (T1w/T2w/FLAIR) | pre-implant, post-surgery | NIfTI (.nii.gz) | `sub-XX_ses-preimplant_T1w.nii.gz` | Manufacturer, MagneticFieldStrength, RepetitionTime, EchoTime, FlipAngle, SliceThickness | DICOM strip + facial defacing required |
| CT | post-implant | NIfTI (.nii.gz) | `sub-XX_ses-postimplant_ct.nii.gz` | Manufacturer, AcquisitionVoltage, SliceThickness, ConvolutionKernel | DICOM strip; defacing recommended |
| EEG (scalp) | pre-implant | EDF or BrainVision | `sub-XX_ses-preimplant_task-XX_eeg.edf` | SamplingFrequency, EEGReference, PowerLineFrequency + electrodes.tsv, channels.tsv | Remove patient name from EDF header |
| iEEG (sEEG) | post-implant | EDF or BrainVision | `sub-XX_ses-postimplant_task-XX_ieeg.edf` | SamplingFrequency, iEEGReference, ElectrodeManufacturer, iEEGPlacementScheme + electrodes.tsv, channels.tsv | Remove patient IDs; coordinates must not encode identifying info |
| DWI | pre-implant | NIfTI + .bval + .bvec | `sub-XX_ses-preimplant_dwi.nii.gz` | Manufacturer, MagneticFieldStrength, RepetitionTime, EchoTime, PhaseEncodingDirection | DICOM strip; defacing not required |
| MR Angiography (TOF) | pre-implant | NIfTI (.nii.gz) | `sub-XX_ses-preimplant_angio.nii.gz` | Manufacturer, MagneticFieldStrength, RepetitionTime, EchoTime, FlipAngle | DICOM strip; defacing recommended |
| fMRI (BOLD) | pre-implant | NIfTI (.nii.gz) | `sub-XX_ses-preimplant_task-rest_bold.nii.gz` | Manufacturer, MagneticFieldStrength, RepetitionTime, EchoTime, TaskName, SliceTiming | DICOM strip; defacing recommended |
| Perfusion (ASL) | pre-implant | NIfTI (.nii.gz) | `sub-XX_ses-preimplant_asl.nii.gz` | Manufacturer, MagneticFieldStrength, RepetitionTime, EchoTime, ArterialSpinLabelingType, PostLabelingDelay | DICOM strip; defacing recommended |
| Field Map | pre-implant | NIfTI (.nii.gz) | `sub-XX_ses-preimplant_magnitude1.nii.gz`, `..._magnitude2.nii.gz`, `..._phasediff.nii.gz` | Manufacturer, MagneticFieldStrength, EchoTime1, EchoTime2, IntendedFor | DICOM strip; defacing not required |

**Localizer and scout scans:** Localizer and scout acquisitions are recognized by the organization tool but are excluded from the shared dataset. They are acquisition aids used to plan the diagnostic scans, not analyzable data, and are not part of a BIDS dataset.

**File compression:** The required imaging format is gzip-compressed NIfTI (`.nii.gz`). Uncompressed `.nii` files submitted to the organization tool are automatically compressed to `.nii.gz` on export, so this requirement is enforced by the tool without a separate manual compression step.

**Repeated acquisitions:** When a subject and session contains more than one acquisition of the same modality (for example two T2w scans, or a TOF angiography scan together with its derived reconstructions), each acquisition is given a BIDS `run-` entity (`run-1`, `run-2`, and so on) so that every file has a unique name. The organization tool assigns these entities automatically.

**Field maps:** A gradient-echo field map produces two magnitude images and one phase-difference image. These are named with the standard BIDS field-map suffixes (`magnitude1`, `magnitude2`, `phasediff`) rather than a single shared suffix, so each image is individually addressable per the BIDS specification. The `EchoTime1` and `EchoTime2` values required in the JSON sidecar accompany the `phasediff` image.

## 4. Metadata Completeness Requirements

Required dataset-level files (per FAIR and ALCOA+):

| Required File | Contents |
|---|---|
| dataset_description.json | Name, BIDSVersion, DatasetType, Authors, GeneratedBy |
| participants.tsv / .json | participant_id only (NeuroGate-generated scaffold); sites complete demographic fields (age, sex, handedness) locally from REDCap and upload the finished file. PHI is never entered into NeuroGate. The accompanying participants.json data dictionary references REDCap as the authoritative source for all demographic fields. |
| README | Dataset overview, contact, collection protocols, known issues |
| CHANGES | Version history of dataset changes per BIDS format |

Each acquisition must include a JSON sidecar with fields per Section 3. EEG/iEEG require electrodes.tsv and channels.tsv with no empty required fields.

## 5. Roles and Responsibilities

| Role | Responsibilities | Accountability |
|---|---|---|
| Data Governance Lead | Maintains this framework; reviews SOP changes; conducts compliance audits; manages QMS | Overall framework integrity; reports to PI/advisor |
| Site Data Manager | Executes SOPs; performs de-identification and BIDS conversion; uploads exported BIDS data to the site's chosen data infrastructure; maintains local audit records | Data quality at site level; completes onboarding |
| Site PI | Ensures local IRB compliance; approves data sharing; maintains PHI key security; signs data use agreements | Institutional compliance; designates data manager |
| REDCap Administrator | Manages project structure and data dictionary; manages user access; monitors entry quality | Metadata integrity and platform management |
| Quality Auditor | Periodic compliance checks; reviews BIDS validation reports; verifies de-identification; flags non-conformances | Independent verification of data quality |

## 6. Audit Traceability

Every dataset must have a documented provenance chain: original acquisition (DICOM), then conversion (dcm2niix), then de-identification, then BIDS organization, then validation, then upload. Each step must be timestamped and attributable. Required audit records:

| Record | Contents | Location |
|---|---|---|
| Conversion Log | dcm2niix command, version, parameters, file counts | Local site records |
| Defacing Log | Tool, version, files processed, QA result | Local site records |
| BIDS Validation | Full validator output (pass/fail, warnings, errors) | Local archive (and data infrastructure if it accepts attachments) |
| Upload Log | Sharing platform upload log: files, timestamps, status | Sharing platform (typically automatic) |
| REDCap Audit Trail | All entries, modifications, user, timestamp | REDCap (automatic) |
| PHI Clearance | Signed confirmation of de-identification completion | Local site + project archive |

## 7. Compliance Verification

### 7.1 Pre-Upload Checklist (Per Subject)

The following must be verified before each upload:

- BIDS validator reports zero errors (warnings acceptable if documented)
- All required metadata files present and complete (JSON sidecars, participants.tsv, electrodes/channels.tsv)
- De-identification completed: DICOM headers stripped, defacing applied (MRI), file names coded, dates removed/shifted, EEG headers cleaned
- PHI Clearance Form signed
- File sizes consistent with expected data volumes

### 7.2 Periodic Audits

Quarterly audits shall review a sample of uploaded datasets. Non-conformances are classified as Critical (data integrity/PHI risk), Major (incomplete metadata/missing records), or Minor (formatting/documentation gaps). Corrective actions are tracked to closure.

### 7.3 Site Onboarding Verification

Before sharing data through the initiative, each site must: designate and train a site data manager, obtain local IRB approval, sign a data use agreement, successfully upload and validate a test dataset, and demonstrate de-identification procedures.
